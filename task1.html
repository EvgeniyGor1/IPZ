Вопрос №1. В чём разница между пре- и пост-коммит Code Reviews? Преимущества и недостатки.

Pre-commit review — при данной технике код попадает в репозиторий только после его апрува на ревью. Код для ревью собирается в патч и заливается на ревью боард, также многие IDE уже имеют функции по шарингу кода на боард

Из плюсов: чистый репозиторий.

Из минусов: обычно, если мы говорим о PHP и WEB, многие разработчики используют привычные им IDE, поэтому автоматизировать процесс ревьювинга при таком подходе очень сложно. Также после успешного прохождения ревью автор может внести в код дополнительные изменения.

Post-commit review — при этом подходе ревью сразу попадает в репозиторий (идеально, если есть специальный бранч для ревьювинга) и потом мерджится в основную ветку.

Из плюсов: свобода для автоматизации процесса, экономия времени на подготовке кода для отправки на ревью, кроссплатформенность и независимость от IDE для решения задач автоматизации.

Из минусов: требуются дополнительные ресурсы для сервера контроля версий, т. к. нагрузка на него несколько увеличивается.
